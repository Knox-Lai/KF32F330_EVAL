# Smart Driver通信接口协议详细解析

## 1. 协议概述

该协议是为智能驱动应用设计的点对点通信协议，用于 \*\* 通信接口模块（CIM）**与**驱动控制模块（DCM）\*\* 之间的通信，采用全双工串行通信链路，通信链路仅包含两个通信伙伴。

协议基于消息实现，常规消息交互采用**请求 - 响应**的通信方式，特殊的异步通知消息采用单向请求方式，无需返回响应。

- 常规场景下，由 CIM 作为请求方向 DCM 发送请求，DCM 作为接收方处理请求并返回响应；
- 协议为对等通信模式，没有主从设备的区分，任意设备都可以作为请求方发起通信，DCM 作为请求方时通常用于向 CIM 发送异步通知。

## 2. 相关定义

### 2.1 核心设备定义

| 术语                                             | 说明                                                                                                                                                               |
| ------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **CIM（Communications Interface Module）** | 通信接口模块，实现外部通信接口（如 DMX、BLE），可以是外部加密狗、内部子卡，也可以直接焊接在驱动的 PCB 上。该模块包含微控制器和实现对应外部通信接口所需的专用电路。 |
| **DCM（Driver Control Module）**           | 驱动控制模块，用于控制一个或多个 LED，通常是驱动中的微控制器及其支持电路，负责控制 LED 的亮度、颜色等参数，同时上报设备状态信息。                                  |
| **Requestor（请求方）**                    | 向其他设备发起请求的设备。                                                                                                                                         |
| **Receiver（接收方）**                     | 接收其他设备请求的设备。                                                                                                                                           |

### 2.2 功能相关定义

| 术语                         | 说明                                                                                                 |
| ---------------------------- | ---------------------------------------------------------------------------------------------------- |
| **Channel（通道）**    | LED 链的单个控制单元，负责单条 LED 链的开关控制和电流调节。                                          |
| **Output（输出单元）** | 实现高级控制功能的通道集合，例如 RGBTW 输出单元包含 5 个通道，通过协同控制实现所需的光强和颜色效果。 |
| **Slot（插槽）**       | 用于标识 DCM 和 CIM 设备的寻址方案，DCM 固定为插槽 0。                                               |
| **Driver（驱动设备）** | 提供照明灯具功率调节和控制功能的设备，包含一个 DCM，可同时支持多个 CIM 接口。                        |

### 2.3 数值与数据类型

#### 2.3.1 数值表示规则

| 数值类型 | 表示方式             | 示例       |
| -------- | -------------------- | ---------- |
| 十进制   | 直接使用数字，无前缀 | `1`      |
| 十六进制 | 以 `0x`为前缀      | `0x01`   |
| 二进制   | 以 `0b`为前缀      | `0b0001` |

#### 2.3.2 数据类型定义

| 类型      | 描述                                                                    |
| --------- | ----------------------------------------------------------------------- |
| U8        | 无符号 8 位数值                                                         |
| U16       | 无符号 16 位数值                                                        |
| U32       | 无符号 32 位数值                                                        |
| U8[]      | U8 类型的数组                                                           |
| Bit-Field | 位域类型                                                                |
| String    | 字符数组，采用 7 位 ASCII 编码，以 null 结尾，未使用的字节需填充为 0x00 |
| Bool      | 布尔值，基于 U8 类型，0 表示 False，1-255 表示 True                     |

#### 2.3.3 字节序规则

多字节数值采用 \*\* 大端（Big-Endian）\*\* 字节序进行编码。

## 3. 通信接口

### 3.1 物理接口参数

协议基于串行 UART 接口实现，具体参数如下：

- 波特率：115200
- 数据位：8
- 奇偶校验：无
- 停止位：1
- 硬件握手：无
- 信号：仅需 TX 和 RX 信号，无需流控制信号

### 3.2 超时与重试机制

- 超时时间：默认 1 秒，请求方等待响应时使用该超时时间
- 重试次数：默认 2 次
  请求方在以下场景可重新发送请求：

1. 等待响应时发生超时
2. 收到错误响应消息，且错误码指示重试可能有效（具体根据命令码和上下文判断）
   部分命令码可自定义超时时间和重试次数。

### 3.3 通信流程（请求 - 响应）

1. 请求方向接收方发送请求消息，进入等待响应状态
2. 接收方监听请求消息，收到请求后进行处理：

   - 若请求处理成功，返回包含所需信息的响应消息
   - 若处理过程中发生错误，返回包含对应错误码的错误响应消息
3. 请求方收到响应后进行对应处理；若遇到错误或超时，可选择重试发送请求

## 4. 通信数据包结构

数据包的结构如下表所示：

| 索引 | 字段     | 字节数 | 类型 | 说明                                                   |
| ---- | -------- | ------ | ---- | ------------------------------------------------------ |
| 1    | 起始码   | 1      | U8   | 固定值 `0xF1`，用于标识数据包的开始                  |
| 2    | 协议版本 | 1      | U8   | 固定值 `1`，标识数据包的结构格式和解释规则           |
| 3    | 标志位   | 2      | 位域 | 包含插槽、数据包类型等信息，定义见下文                 |
| 4    | 事务 ID  | 1      | U8   | 递增数值，用于匹配请求和响应，取值范围 0-255，循环使用 |
| 5    | 命令码   | 2      | U16  | 标识请求的具体操作，详细定义见「9. 命令码定义」        |
| 6    | 负载长度 | 2      | U16  | 负载内容的字节长度                                     |
| 7    | 负载     | 0-538  | U8[] | 可选字段，内容由对应的命令码决定                       |
| 8    | 校验和   | 2      | U16  | 对索引 2 到索引 7 的内容计算得到                       |
| 9    | 停止码   | 1      | U8   | 固定值 `0xF2`，用于标识数据包的结束                  |

- 数据包总长度：`12 + 负载长度`
- 最大负载大小：538 字节
- 最大数据包大小：550 字节

### 4.1 各字段详细说明

#### 4.1.1 起始码

固定值 `0xF1`，仅能作为数据包的起始标识使用，若该数值需要在数据包的其他位置出现，需进行转义处理（见「8. 保留字节转义」）。

#### 4.1.2 协议版本

固定值 `1`，用于标识数据包的结构格式和字段解释规则。若未来对数据包结构或字段含义进行破坏性修改，该数值会相应增加。
接收方需要校验该版本，若不支持该版本的协议，则直接丢弃数据包。

#### 4.1.3 标志位

标志位为 2 字节的位域，结构如下：

| 位 15-6                       | 位 5-3                                             | 位 2-0             |
| ----------------------------- | -------------------------------------------------- | ------------------ |
| 保留位（必须设置为 0）        | Slot（插槽标识，当前固定为 0，未来用于多插槽寻址） | Type（数据包类型） |
| 其中 `Type`字段的取值说明： |                                                    |                    |

| Type 值 | 数据包类型     | 说明                     |
| ------- | -------------- | ------------------------ |
| 0       | 请求数据包     | 请求方发送的请求消息     |
| 1       | 响应数据包     | 接收方返回的正常响应消息 |
| 2       | 错误响应数据包 | 接收方返回的错误响应消息 |
| 3-7     | 保留           | 未来扩展使用             |

#### 4.1.4 事务 ID

由请求方设置的递增数值，接收方在返回响应时，需要使用与请求消息相同的事务 ID，用于匹配请求和响应。

#### 4.1.5 命令码

16 位的数值，分为两个位域：

- 组（高 8 位）：用于将相关的命令归类，便于管理和扩展
- 方法（低 8 位）：组内的具体操作，在组内唯一

#### 4.1.6 负载长度

标识负载内容的字节长度，具体数值由对应的命令码和响应类型决定。

#### 4.1.7 负载

负载的内容由对应的命令码决定，不同的命令码对应不同的负载结构和内容。

#### 4.1.8 校验和

采用**CRC-16/CCITT-FALSE**算法计算，算法参数如下：

- 宽度：16 位
- 多项式：`0x1021`（对应公式：`x^16+x^12+x^5+1`）
- 初始值：`0xFFFF`
- 输出反射：False
- 输出异或：`0x0000`
- 校验值：`0x29B1`

#### 4.1.9 停止码

固定值 `0xF2`，仅能作为数据包的结束标识使用，若该数值需要在数据包的其他位置出现，需进行转义处理（见「8. 保留字节转义」）。

## 5. 请求消息

请求消息基于上述的通信数据包结构构建，构建时需要遵循以下规则：

1. 设置协议版本为当前的协议版本（固定值 1）
2. 配置标志位：Slot 设置为 0，Type 设置为 0（请求数据包）
3. 设置事务 ID 为下一个序列数值
4. 设置命令码为所需的操作对应的命令码
5. 根据命令码的定义，设置负载长度和负载内容
6. 计算校验和并添加到数据包中

接收方在收到请求消息后，需要进行以下验证：

1. 验证计算得到的校验和与数据包中的校验和是否匹配，若不匹配则返回错误响应消息，错误码为 `ERR_PACKET_CHECKSUM`
2. 验证实际的负载长度与数据包中指定的负载长度是否一致，若不一致则返回错误响应消息，错误码为 `ERR_PACKET_LENGTH`
3. 验证命令码是否被接收方支持，若不支持则返回错误响应消息，错误码为 `ERR_UNSUPPORTED_COMMAND`
4. 验证负载长度是否符合该命令码的要求，若不符合则返回错误响应消息，错误码为 `ERR_PAYLOAD_LENGTH`

## 6. 响应消息

响应消息基于上述的通信数据包结构构建，用于向请求方返回请求的处理结果，构建时需要遵循以下规则：

1. 设置协议版本：可设置为当前的协议版本，或设置为请求消息中的协议版本（若接收方支持该版本，需按照该版本的协议格式构建整个响应消息）
2. 配置标志位：Slot 设置为 0，Type 设置为 1（响应数据包）
3. 设置事务 ID 为请求消息中的事务 ID
4. 设置命令码为请求消息中的命令码
5. 根据该命令码的响应定义，设置负载长度和负载内容
6. 计算校验和并添加到数据包中

接收方（即原请求方）在收到响应消息后，需要进行以下验证：

1. 验证计算得到的校验和与数据包中的校验和是否匹配
2. 验证实际的负载长度与数据包中指定的负载长度是否一致
3. 验证命令码是否与对应的请求消息的命令码匹配
4. 验证事务 ID 是否与对应的请求消息的事务 ID 匹配
5. 验证负载长度是否符合该命令码响应的要求

若验证失败，可选择以下处理方式：

1. 直接丢弃该响应消息
2. 记录错误信息或通知相关人员
3. 重试发送对应的请求消息
   部分命令码会定义特定的错误处理方式。

## 7. 错误响应消息

错误响应消息用于向请求方通知请求处理过程中发生的错误，基于上述的通信数据包结构构建，构建时需要遵循以下规则：

1. 设置协议版本：可设置为当前的协议版本，或设置为请求消息中的协议版本（若接收方支持该版本，需按照该版本的协议格式构建整个错误响应消息）
2. 配置标志位：Slot 设置为 0，Type 设置为 2（错误响应数据包）
3. 设置事务 ID 为请求消息中的事务 ID
4. 设置命令码为请求消息中的命令码
5. 设置负载长度和负载内容，负载的结构如下：| 索引 | 字段     | 字节数 | 类型   | 说明                                             |
   | ---- | -------- | ------ | ------ | ------------------------------------------------ |
   | 1    | 错误码   | 4      | U32    | 标识错误的类型，详细定义见「11. 标准错误码」     |
   | 2    | 描述长度 | 2      | U16    | 错误描述文本的字节长度                           |
   | 3    | 错误描述 | 0-512  | String | 可选的错误描述文本，用于提供额外的错误上下文信息 |
6. 计算校验和并添加到数据包中

接收方（即原请求方）在收到错误响应消息后，需要进行以下验证：

1. 验证计算得到的校验和与数据包中的校验和是否匹配
2. 验证实际的负载长度与数据包中指定的负载长度是否一致
3. 验证命令码是否与对应的请求消息的命令码匹配
4. 验证事务 ID 是否与对应的请求消息的事务 ID 匹配
5. 验证负载长度是否符合错误响应消息的要求

若验证失败，可选择以下处理方式：

1. 直接丢弃该错误响应消息
2. 记录错误信息或通知相关人员
3. 重试发送对应的请求消息
   部分命令码会定义特定的错误处理方式。

## 8. 保留字节转义

### 8.1 保留字节列表

以下字节为保留字节，仅能在指定的位置使用：

| 字节值 | 用途         |
| ------ | ------------ |
| 0xF0   | 未来扩展使用 |
| 0xF1   | 数据包起始码 |
| 0xF2   | 数据包停止码 |
| 0xF3   | 转义码       |

### 8.2 字节转义规则

若上述保留字节需要出现在数据包的索引 2 到索引 7 的位置，需要使用转义技术将其转换为两字节的序列：

- 转义操作：将保留字节转换为 `0xF3` + (原字节 & `0x0F`)
- 反转义操作：移除转义码 `0xF3`，将第二个字节与 `0xF0`进行或运算，得到原字节值

示例：若需要在数据包的非起始码位置使用 `0xF1`，则需要将其转义为 `0xF3 0x01`；接收方在处理时，将 `0x01`与 `0xF0`进行或运算，得到原字节 `0xF1`。

## 9. 命令码定义

命令码是 16 位的数值，由 \*\* 组（高 8 位）**和**方法（低 8 位）\*\* 两个位域组成，组用于将相关的命令归类，便于管理和扩展。

### 9.1 命令组定义

| 组值      | 组名称            | 说明                                               |
| --------- | ----------------- | -------------------------------------------------- |
| 0x00      | 操作组            | 用于控制设备正常运行的命令，如亮度调节、色温调节等 |
| 0x08      | 发现组            | 用于设备发现和功能标志管理的命令                   |
| 0x10      | 属性 / 配置文件组 | 用于设备配置管理的命令                             |
| 0x18      | OTA 组            | 用于 OTA 固件更新的命令                            |
| 0x20      | 遥测组            | 用于传感器管理的命令，如 LED 寿命、温度数据获取等  |
| 0x28      | 调试 / 测试组     | 用于测试和调试的命令                               |
| 0x30      | 异步通知组        | 单向请求消息，无需返回响应                         |
| 0x38      | 临时开发组        | 仅用于临时开发和测试，无需在协议文档中注册         |
| 0x40-0xFF | 保留组            | 未来扩展使用                                       |

### 9.2 主要命令说明

#### 9.2.1 发现组命令

| 命令码 | 命令名称           | 说明                   |
| ------ | ------------------ | ---------------------- |
| 0x0800 | Identify Discovery | 设备发现命令           |
| 0x0801 | Get Features List  | 获取设备支持的功能列表 |
| 0x0802 | Get Feature Detail | 获取指定功能的详细信息 |
| 0x0803 | Get Command List   | 获取设备支持的命令列表 |

#### 9.2.2 属性 / 配置文件组命令

| 命令码 | 命令名称            | 说明               |
| ------ | ------------------- | ------------------ |
| 0x1000 | Get Properties List | 获取设备的属性列表 |
| 0x1001 | Get Property        | 获取指定属性的信息 |
| 0x1002 | Set Property        | 设置指定属性的数值 |

#### 9.2.3 OTA 组命令

| 命令码 | 命令名称      | 说明                                                                                                                                                                  |
| ------ | ------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 0x1800 | Reboot        | 重启设备，可指定重启延迟时间；若延迟时间为 0，则中止当前的重启倒计时                                                                                                  |
| 0x1801 | Begin OTA     | 启动 OTA 固件更新会话，需要提供固件的 MD5 哈希值、固件长度、固件 ID、固件版本等信息，可设置降级标志（允许安装同版本或旧版本固件）和强制标志（忽略固件 ID 和版本验证） |

| 命令码 | 命令名称 | 说明                                             |
| ------ | -------- | ------------------------------------------------ |
| 0x2000 | 异步通知 | 传感器数据的异步通知，属于单向请求，无需返回响应 |

## 10. CRC 实现定义

协议使用**CRC-16/CCITT-FALSE**算法计算校验和，算法的详细参数如下：

- 通用名称：CRC-16/CCITT-FALSE
- 别名：CRC-16/IBM-3740、CRC-16/AUTOSAR
- 宽度：16 位
- 多项式：`0x1021`（对应公式：`x^16+x^12+x^5+1`）
- 初始值：`0xFFFF`
- 输出反射：False
- 输出异或：`0x0000`
- 校验值：`0x29B1`

## 11. 标准错误码

| 错误码 | 错误标签                   | 描述                                                 |
| ------ | -------------------------- | ---------------------------------------------------- |
| 0      | ERR_UNKNOWN                | 未知错误，无法归类的错误类型                         |
| 1      | ERR_PROTOCOL_VERSION       | 不支持的协议版本，接收方无法处理该版本的数据包       |
| 2      | ERR_PACKET_CHECKSUM        | 校验和验证失败，数据包可能被篡改或传输错误           |
| 3      | ERR_PACKET_LENGTH          | 无效的数据包长度，实际数据包长度与标识的长度不符     |
| 4      | ERR_UNSUPPORTED_COMMAND    | 不支持的命令码，接收方无法识别该命令                 |
| 5      | ERR_PAYLOAD_LENGTH         | 负载长度与命令码不匹配，负载长度不符合该命令的要求   |
| 6      | ERR_OTA_UNSUPPORTED_TARGET | 不支持的 OTA 目标，指定的 OTA 目标设备不被支持       |
| 7      | ERR_OTA_UNSUPPORTED_REGION | 不支持的 OTA 区域，指定的固件存储区域不被支持        |
| 8      | ERR_OTA_IMAGE_OVERSIZE     | OTA 镜像过大，超出了目标设备存储区域的容量           |
| 9      | ERR_OTA_INCOMPATIBLE_ID    | 固件 ID 不兼容，该固件无法在目标设备上运行           |
| 10     | ERR_OTA_OLD_VERSION        | OTA 固件版本低于当前设备安装的版本，且未设置降级标志 |
| 11     | ERR_OTA_NOT_ACTIVE         | OTA 会话未激活，需要先执行 Begin OTA 命令            |
